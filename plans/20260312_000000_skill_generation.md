# Skill Generation for Host Agents

## Goal
When bab starts, installs/removes plugins, or runs `bab onboard`, generate Agent Skills (open standard: https://agentskills.io) for the host agent describing bab's MCP tools and installed delegate plugins.

---

## Context
- Bab exposes 17 MCP tools + delegate plugins via MCP protocol
- Agent Skills is an open standard supported by Claude Code, OpenAI Codex, OpenCode, Cursor, Gemini CLI, Copilot, Goose, and 20+ other agents
- No mechanism exists today to tell the host agent *when* and *how* to use bab's tools
- Skills must stay in sync with installed plugins and bab version

## Agent Skills Standard

Ref: https://agentskills.io/specification

A skill is a **directory** containing a `SKILL.md` file (required) + optional supporting files:
```
skill-name/
├── SKILL.md           # Required: YAML frontmatter + markdown body
├── references/        # Optional: detailed docs loaded on demand
├── scripts/           # Optional: executable code
└── assets/            # Optional: templates, resources
```

### SKILL.md format
```yaml
---
name: skill-name          # Required. Lowercase, hyphens, max 64 chars. Must match dir name.
description: ...          # Required. Max 1024 chars. What it does + when to use it.
license: MIT              # Optional.
compatibility: ...        # Optional. Environment requirements.
metadata:                 # Optional. Arbitrary key-value.
  author: bab
  version: "1.0.0"
allowed-tools: ...        # Optional. Pre-approved tools (experimental).
---

Markdown body with instructions...
```

### Claude Code extensions (beyond standard)
- `disable-model-invocation: true` — prevent auto-loading (user-only invoke)
- `user-invocable: false` — hide from / menu (agent-only background knowledge)
- `context: fork` — run in subagent
- `argument-hint` — autocomplete hint

### Progressive disclosure
1. `name` + `description` (~100 tokens) — loaded at startup for all skills
2. SKILL.md body (< 5000 tokens recommended) — loaded when skill activates
3. references/ files — loaded only when needed

Keep SKILL.md under 500 lines. Move detailed reference material to separate files.

## Agent Skill Directories

| Agent | Config dir | Skills dir |
|-------|-----------|------------|
| Claude Code | `~/.claude/` | `~/.claude/skills/` |
| OpenAI Codex | `~/.codex/` | `~/.codex/skills/` |

Note: OpenCode skills dir needs verification before adding (may be `~/.config/opencode/` not `~/.opencode/`). Add only after confirming.

Discovery: check if config dir exists. Only write skills for detected agents.
Do NOT create the agent config dir — only create the `skills/` subdir if the parent exists.

## Generated Skill Structure

Bab generates **one skill directory** per agent skills dir:

```
<agent-skills-dir>/
└── bab/
    ├── SKILL.md                    # Overview: what bab is + when to use each tool
    ├── .bab-skills.json            # Staleness metadata (not part of the standard)
    └── references/
        ├── tools.md                # Detailed tool reference (all MCP tools)
        └── delegate-plugins.md     # Delegate plugin details, roles, examples
```

### SKILL.md content
```yaml
---
name: bab
description: Bab MCP server tools for AI-assisted development. Use for code analysis, debugging, code review, planning, multi-model consensus, security audits, test generation, documentation, and delegating tasks to CLI agents (Claude Code, Codex, OpenCode, Copilot). Activate when working on complex coding tasks that benefit from structured workflows or second opinions.
user-invocable: false
metadata:
  author: bab
  version: "<bab_version>"
  generated-at: "<timestamp>"
---

<!-- Auto-generated by bab v<version> — do not edit manually -->
<!-- Regenerate with: bab onboard -->

# Bab MCP Tools

<tool overview table: name, when to use, key params>

For detailed tool reference, see [references/tools.md](references/tools.md).

# Delegate Plugins

<plugin overview table: name, CLI, roles>

For plugin details and usage examples, see [references/delegate-plugins.md](references/delegate-plugins.md).
```

- `user-invocable: false` — bab is background knowledge, not a slash command
- Description includes keywords for all major capabilities so agents match it to relevant tasks

### references/tools.md
For each MCP tool:
- Name, full description
- Required parameters with types
- Usage example
- When to use vs when NOT to use

### references/delegate-plugins.md
For each loaded plugin (bundled + installed):
- Plugin name, ID, underlying CLI command
- Available roles with descriptions
- Capabilities (images, streaming, working directory)
- Usage example: `delegate(cli_name: "<id>", prompt: "...", role: "<role>")`
- Link to docs

## Staleness Detection

### Metadata (`.bab-skills.json`)
```json
{
  "schema_version": 1,
  "bab_version": "1.0.0",
  "generated_at": "2026-03-12T00:00:00Z",
  "fingerprint": "<sha256 hash>"
}
```

Fingerprint = SHA-256 of `${bab_version}:${sorted_plugin_ids}:${sorted_tool_names}`.
Single string comparison for staleness check.

### User-modified detection
Only overwrite SKILL.md files that contain the auto-generated header marker:
```
<!-- Auto-generated by bab v
```
If SKILL.md exists but lacks the marker → user-modified, skip with warning.

## Triggers

### 1. `bab onboard` (explicit CLI command)
- Always regenerates, ignoring staleness
- Discovers agents, generates skills for all detected (or `--agent <name>` for one)
- Prints summary: which agents updated, how many tools/plugins

### 2. `bab add` (after successful install)
- Call skill regeneration after `installPluginsFromSource()` returns
- Respects staleness check (but will always be stale since plugins changed)

### 3. `bab remove` (after successful removal)
- Call skill regeneration after plugin directory is deleted

### 4. Server startup (auto-check)
- After `registerCoreTools()` + `loadConfig()`, after `server.connect()`
- Check staleness → regenerate only if stale
- Fire-and-forget: catch errors, log warning to stderr, don't block server
- All output to stderr (stdout reserved for MCP protocol)

## Public API

```typescript
async function regenerateSkills(
  config: BabConfig,
  options?: {
    force?: boolean;       // skip staleness check (bab onboard)
    agent?: string;        // target specific agent
    stderr?: WritableLike; // for warnings, defaults to process.stderr
  },
): Promise<void>
```

`config` is the only required param. The function internally discovers agents, loads plugins, generates content, checks staleness, and writes. Callers don't need to know about any of that.

## Tool Catalog

The generator accepts `toolNames: string[]` so:
- Server startup path passes live names: `Array.from(server.toolRegistry.keys())`
- CLI paths (onboard, add, remove) pass a static list from a constant

Add a startup assertion in `server.ts`: verify registered tool names match the static list to catch drift early.

## Atomic Directory Replacement

`rename()` can't atomically replace a non-empty directory. The safe pattern:
1. Build new skill dir in `<skills-dir>/.bab-staging-<nonce>/`
2. If existing `bab/` dir: rename to `<skills-dir>/.bab-old-<nonce>/`
3. Rename staging → `bab/`
4. Delete old dir
5. On failure: restore old dir if it was moved, clean up staging

## Error Handling
- Agent config dir doesn't exist → skip silently
- Skills dir doesn't exist → create it (parent agent config dir must exist)
- SKILL.md exists without bab header → skip with stderr warning
- Write failure → log warning, continue (non-fatal)
- Replacement failure → restore old dir, clean up staging

## Security
- Never include API keys or env values in skill files
- Skill files contain only public tool/plugin metadata
- Atomic writes prevent partial/corrupt skill files

## Resolved Questions
1. **Verify agent MCP config?** No — bab won't work if the config is wrong anyway. Out of scope.
2. **`--agent <name>` flag?** Yes — useful when running `bab onboard` directly from CLI outside an agent session.
3. **Example prompts in skills?** Yes — include examples + link to docs. Docs are the source of truth; skills reference them.
4. **One file or directory?** Directory per the Agent Skills open standard. `bab/SKILL.md` + `bab/references/`.
5. **Agent-specific content?** No. Same standard content for all agents. Only the destination path differs.
6. **Standard-only or Claude Code extensions?** Use standard fields + Claude Code `user-invocable: false`. Other agents ignore unknown frontmatter.
