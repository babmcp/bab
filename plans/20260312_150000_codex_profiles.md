# Plan: Codex profile support in delegate plugin

## Goal

Let users specify a Codex configuration profile (`-p/--profile`) when delegating to Codex via bab.

## Context

Codex CLI supports `-p, --profile <CONFIG_PROFILE>` which loads a named profile from `~/.codex/config.toml`. Profiles can set model, sandbox mode, approval policy, features, etc.

Current Codex adapter (`bab-plugins/codex/adapter.ts`):
- Builds: `codex exec --json --enable web_search_request [role args] <prompt>`
- `ALLOWED_FLAGS`: `model`, `temperature`, `max-tokens`, `enable`
- Does NOT support `--profile`

## Design

### Add `profile` to adapter's ALLOWED_FLAGS

One-line change. The existing flag-to-arg loop already handles string values → appends `--profile <value>` to the command.

### Profile specified via role args (already works)

Users can set profile per-role in `manifest.yaml`:

```yaml
roles:
  - name: researcher
    description: Research-focused Codex role using fast profile.
    prompt_file: prompts/default.txt
    args:
      profile: fast
```

Or pass it at MCP tool-call time via `role.args.profile` — both paths flow through `buildCommand` → `ALLOWED_FLAGS` check → `--profile <value>`.

### What we are NOT doing (and why)

- **NOT adding `profile` to the generic delegate tool schema** — profile is Codex-specific. Adding it to the generic tool leaks Codex knowledge into the shared delegate interface.
- **NOT adding bab config-file-level defaults** (`plugins.codex.defaults.profile`) — there's no `config.yaml` today, no plugin config merging in adapter-wrapper, and adapter-wrapper doesn't have access to BabConfig. High cost, near-zero benefit for a single flag. Can revisit if multiple plugins need config defaults.

## Changes needed

### `bab-plugins/codex/adapter.ts`
- Add `"profile"` to `ALLOWED_FLAGS` set

That's it.

### Optional: `bab-plugins/codex/manifest.yaml`
- Add example role with profile arg to demonstrate usage

## Steps

1. Add `"profile"` to `ALLOWED_FLAGS` in `bab-plugins/codex/adapter.ts`
2. Add test: profile arg flows through to codex command
3. Optionally add a role example in manifest.yaml
